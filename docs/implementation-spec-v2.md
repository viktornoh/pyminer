## 1) 범위 정의 — Must / Should / Won’t

## Must (오늘 14:30까지 반드시 완료)
1. **코어 루프 동작**
   - 라운드 시작(3초 카운트다운) → 플레이 진행 → 라운드 종료 → 결과 화면까지 1회 무중단 재생
2. **기본 플레이 상태 표시(HUD)**
   - 남은 시간, HP, 점수 표시
3. **채팅 명령 최소 4개 적용**
   - `tnt`, `boost`, `slow`, `shield` (혹은 동등한 4개)
   - 유저별 쿨다운(10~20초 범위), 전역 큐 처리 주기(2~5초 범위) 적용
4. **후원사 스킬 3개 구현**
   - 발동 조건 또는 쿨다운 기반
   - 활성 시 이름/효과 문구가 HUD 또는 이벤트 영역에 표시
5. **결과 화면 MVP**
   - 총점, Top 참여자(최소 1~3명), 스킬 기여 요약(발동 횟수)
6. **설정/밸런스 외부화**
   - `config.json`으로 핵심 파라미터(쿨다운, 지속시간, 점수 배수 등) 조정 가능
7. **운영 안정성 최소선**
   - 실행 실패 시 재실행 가능한 스크립트(`run.sh`) 확인

## Should (오늘 완료 권장, 미완료 시 후속)
1. 9:16 기본 + 16:9 오버레이 호환 레이아웃 점검
2. 최근 채팅 명령 로그(최소 최근 3~5개) 시각화
3. 리허설 모드(자동 입력/봇 이벤트) 1개
4. 간단 로그 저장(라운드 점수/명령 채택/스킬 발동)
5. 행사 진행용 3분 데모 스크립트 초안

## Won’t (이번 타임박스에서 제외)
1. 계정 연동 랭킹/영속 프로필
2. 웹 기반 실시간 운영 대시보드
3. 다국어 운영 UI 완성
4. Twitch/Discord 다중 채팅 연동(YouTube 외 확장)
5. 고도화 아트 폴리싱(게임성 영향 없는 시각 디테일)

## 3) 수용 기준 (Acceptance Criteria)

## 기능 수용 기준
1. **E2E 라운드 성공**
   - Given 게임이 시작되면
   - When 한 라운드를 실행하면
   - Then 크래시 없이 결과 화면까지 전이된다.
2. **채팅 명령 반영 가능**
   - Given 서로 다른 3명 이상의 입력이 들어오면
   - When 큐 처리 주기에 도달하면
   - Then 유효 명령이 규칙(쿨다운/처리량 제한) 내에서 반영된다.
3. **후원사 스킬 동작**
   - Given 스킬 발동 조건이 충족되면
   - When 스킬이 발동하면
   - Then 효과가 설정 시간 동안 적용되고 종료 후 원상복구된다.
4. **결과 데이터 표시**
   - Given 라운드가 종료되면
   - Then 총점, 참여자 기여 정보, 스킬 발동 요약이 화면에 표시된다.

## 품질 수용 기준
1. 3분 데모 중 P0(진행 불가) 버그 0건
2. 목표 해상도(기본 9:16)에서 HUD 텍스트가 읽힌다.
3. 설정값 1개 이상 변경(예: boost 지속시간) 시 재실행 후 실제 반영된다.
4. 네트워크/채팅 입력이 없을 때도 리허설 모드 또는 내부 이벤트로 라운드 시연 가능하다.

## 4) 디자이너/개발자 작업 경계 (R&R)

## 디자이너 책임 (What Designer owns)
1. **정보 우선순위 설계**
   - HUD 계층(시간/HP/점수/활성 스킬/최근 명령) 가독성 우선순위 정의
2. **이벤트 시각 규칙 정의**
   - 후원사 카드 노출 길이(1~2초), 색/아이콘/문구 톤
3. **결과 화면 카피/레이아웃**
   - 총점, Top 참여자, 스킬 기여도의 전달 순서 및 문구
4. **데모 연출 가이드**
   - 사회자 멘트와 화면 변화가 맞물리는 타이밍 가이드

## 개발자 책임 (What Developer owns)
1. **게임 루프/상태머신 구현**
2. **물리/점수/충돌 및 명령 반영 로직 구현**
3. **채팅 큐, 쿨다운, 폭주 방지 로직 구현**
4. **후원사 스킬 시스템(등록/발동/종료) 구현**
5. **결과 집계/로그/실행 스크립트 등 운영성 확보**

## 공동 의사결정 항목 (Joint)
1. 스킬 밸런스 상한(사기성 방지 수치)
2. 데모에서 보여줄 우선 시나리오 1개 선정
3. 시간 부족 시 절삭 순서(Should 항목 중 제외 우선순위)

## 경계 원칙
- 디자이너는 **무엇을 보여줄지(표현/우선순위)**를 결정한다.
- 개발자는 **어떻게 동작시킬지(시스템/성능/안정성)**를 결정한다.
- 인터랙션 영향이 큰 변경(예: 명령 의미 변경, 스킬 효과 변경)은 반드시 공동 검토 후 반영한다.
